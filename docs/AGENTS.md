# Agent Inventory

> **Status**: stable
> **Scope**: Agent roles, capabilities, topology, and memory patterns — authoritative for what each agent does, what it reads/writes, and how agents relate to each other
> **Last updated**: 2026-03-20 (processing triggers + acquisition)

---

## Overview

The pipeline uses eight distinct agent roles. They divide naturally into three tiers:

| Tier | Agents | Characteristic |
|---|---|---|
| **Ingestion** | Source examination, Ingestion worker, Discovery search, Research search | Face the external world; produce raw artifacts |
| **Analysis** | Processing, Beat, Research, Connection | Transform and interpret; read/write the artifact store |
| **Editorial** | Editor | Produce publishable output; interface with humans |

Assignment orchestration sits across all tiers: it is not a separate agent but a coordination layer described at the end.

---

## Source Examination Agent

**Role**: Understand how a specific source works and write a Scrapy spider that will fetch from it on every scheduled run.

**Reads**: The source endpoint (and any pages reachable from it)

**Writes**: A `.py` spider file to `services/ingestion/src/sidekick/spiders/`; does not write to the database

**Tools**:
- `fetch_url` — HTTP client for browsing the source
- `write_spider` — writes and validates a Scrapy spider file (AST parse + `SidekickSpider` subclass check)

**Memory**: Stateless — each examination is independent. The spider file is the persistent artifact.

**Framework**: DeepAgents (LangChain `create_react_agent`) — examining a source involves open-ended browsing where the number of steps is unknown at design time. Model and system prompt configured inline in `examination.py`; no `agent_configs` DB row required (developer-only tool).

**Topology**: Developer-triggered only. Runs once per source; not scheduled. Output is reviewed and committed by a developer before the spider is activated.

**Entrypoint**: `sidekick examine --url URL --beat BEAT --geo GEO [--name NAME]`

**Workflow**:
1. Agent browses the source, follows links, identifies document/media URLs and their patterns
2. Agent calls `write_spider` with a complete `SidekickSpider` subclass
3. Developer reviews the generated `.py` file, tests it, commits it
4. `sidekick spiders sync` upserts the `Source` row in DB (`examination_status=active`)

**Key behaviors**:
- Produces working, runnable code — not a description of what to do
- Yields one `RawItem` per asset (PDF, video, audio), not one per page — if a detail page has both a PDF and a video, the spider yields both
- Spider files import only `scrapy` and `sidekick.spiders._base`; no LLM calls inside a spider
- If the site requires auth or complex JS that can't be reliably automated, the agent notes this in a comment and writes a partial spider for human completion
- Does not create a Source DB row; that happens via `spiders sync` after developer review

## Ingestion Worker (Scrapy Spiders)

**Role**: Fetch new items from known sources on schedule; write raw artifacts to the store. Fully deterministic — no LLM at crawl time.

**Reads**: External sources (via HTTP); artifact store (for deduplication on startup)

**Writes**: `raw` artifacts to the artifact store; health updates back to the source registry

**Implementation**: One `SidekickSpider` subclass per source, generated by the examination agent and committed to `services/ingestion/src/sidekick/spiders/`. The spider harness provides:
- `ArtifactWriterPipeline` — converts `RawItem`s to `Artifact` rows; routes binary content (PDF, audio, video) to object storage, text inline
- `DeduplicationMiddleware` — loads already-ingested URLs from artifact store on `spider_opened`; drops duplicates within and across runs
- `run_spider()` / `run_spiders()` — wires `CrawlerProcess` with infrastructure; updates source health afterward

**Memory**: Stateless per run — dedup state loaded from artifact store at startup, health written back at completion.

**Topology**: Spiders run sequentially via `sidekick spiders due` (cron-based) or on demand via `sidekick spiders run SOURCE_ID`. Can be parallelized later if needed.

**Key behaviors**:
- Runs only for sources with `examination_status: active` (set by `spiders sync`)
- Deduplicates against `source-url` entities in existing `stage=raw` artifacts for the source
- Updates `last_checked`, `last_new_item`, and `status` in the source registry after every run
- Yields one `RawItem` per asset — PDFs, audio, and video are separate items even when found on the same page
- Does not analyze or interpret content — classifies by media type only, passes everything downstream

## Discovery Search Agent

**Role**: Find new recurring sources that are not yet in the registry.

**Reads**: Source registry (to avoid proposing duplicates); external web

**Writes**: Proposed source entries to the registry (flagged for human review, not auto-activated)

**Tools**:
- Web search
- Page fetching and link extraction
- Registry read/write

**Memory**: Light persistent memory useful — can maintain a list of already-explored domains and previous proposals to avoid re-surfacing the same suggestions.

**Topology**: Singleton or small pool. Runs periodically or triggered by an assignment.

**Key behaviors**:
- Operates on prompts like "find all recurring public document sources for Springfield, IL city government"
- Produces source proposals, not live sources — human review or confidence threshold required before activation
- Assigns initial trust level `agent` to all proposals
- Should cluster related sources (e.g., agenda page and minutes page for the same body) into related-source groups

---

## Research Search Agent

**Role**: Fetch specific documents on demand; the targeted fetch complement to ingestion's scheduled fetch.

**Reads**: Nothing in the store; receives a targeted request (URL, query, or description of needed document)

**Writes**: `raw` artifacts tagged with the requesting assignment ID

**Tools**:
- Web search
- HTTP client / PDF download
- Document extraction (to verify content matches the request)

**Memory**: Stateless — fully driven by the request parameters.

**Topology**: Pool. Triggered by assignments (gap analysis identifies a missing document) or directly by beat/research agents requesting source material.

**Key behaviors**:
- Does not run on a schedule — only triggered on demand
- Validates that the fetched document is actually relevant before writing (don't write noise to the store)
- If the source doesn't exist in the registry but proves recurring, surfaces a source proposal as a side effect
- Tags output with the requesting assignment ID for traceability

## Processing Agents

**Role**: Acquire async raw stubs when needed, then transform **complete** raw artifacts into structured, analyzable form.

**Reads**: `raw` artifacts from the artifact store (`active` for processors; `pending_acquisition` for acquisition workers only)

**Writes**: `processed` artifacts (`document-text`, `transcript-clean`, `summary`, `entity-extract`, `structured-data`). Completes async raw stubs via `ArtifactStore.complete_acquisition()` (same row becomes `active` with `content_uri`).

**Triggers**: React to **`artifact_written` only when `stage="raw"` and `status="active"`** for content processors. React to **`acquisition_needed`** for HLS/yt-dlp-style stubs. Ignore `pending_acquisition` on the general processing path.

**Tools**:
- ffmpeg / yt-dlp (async acquisition)
- Speech-to-text / transcription (for audio and video)
- PDF text extraction and OCR
- Document summarization (LLM)
- Named entity recognition (NER)
- Structured data extraction (tables, budgets, vote records from unstructured text)

**Memory**: Stateless — each raw artifact is processed independently.

**Topology**: Multiple specialized instances by media type:
- **Transcript processor**: audio/video → `transcript-clean`
- **Document processor**: PDFs, HTML → `summary` + `entity-extract`
- **Structured data extractor**: budget sheets, vote records → `structured-data`

Could be unified with routing logic, or kept separate to allow independent scaling and model selection.

**Key behaviors**:
- Must declare `derived_from` lineage on every artifact written — enforced at write time
- Produces multiple artifact types per input (e.g., a PDF produces both a `summary` and an `entity-extract`)
- Should flag processing failures explicitly rather than silently producing low-quality output
- Does not interpret meaning — that is beat/research agents' job

## Beat Agents

**Role**: Maintain ongoing expertise on a specific coverage domain; produce analytical briefs from processed artifacts.

**Reads**: `processed` artifacts filtered to their beat and geo; raw artifacts (when summaries lose nuance); `analysis` artifacts from prior runs (their own history)

**Writes**: `beat-brief`, `flag`, `trend-note` artifacts to the analysis layer

**Tools**:
- Artifact store read (structured query + semantic search)
- Lineage traversal (to reach back to raw when needed)
- Research search agent trigger (to request specific missing documents)
- LLM for analysis and writing

**Memory**: **Persistent** — this is the defining characteristic of beat agents. A beat agent carries:
- Running narrative of what's happening on the beat (ongoing stories, recurring actors, pending votes)
- Known entity registry for the beat (council members, department heads, organizations)
- Historical baselines (last year's budget figures, prior vote records, established policies)
- Flags for items that were noted but not yet acted on

Memory is structured, not a raw conversation log. It is queryable state that agents update on each run.

**Topology**: One agent per beat (× geo if beats are geo-specific). Example beats:
- `government:city_council` (per city)
- `education:school_board` (per district/region)
- `housing_zoning:zoning_board` (per city)
- `public_safety:police` (per city)
- `budget_finance` (cross-cutting — see Research Agents below)

Beat agents are long-running processes, not one-shot invocations. They wake up when new processed artifacts arrive on their beat.

**Key behaviors**:
- Maintains context across multiple council meetings, budget cycles, and policy changes — not just a per-document summarizer
- Produces briefs that synthesize multiple processed artifacts into a coherent beat narrative
- Flags items that warrant editorial attention (`flag` artifact type)
- Can request specific documents via the research search agent when the store is missing something needed to complete an analysis
- Can spawn research sub-assignments autonomously (e.g., "I need the full text of ordinance 2026-14 to complete this brief")
- Notified by assignment system when tagged work completes

## Research Agents

**Role**: Track cross-cutting policy and budget topics that don't belong to a single beat.

**Reads**: `processed` artifacts (policy, budget, regulatory documents); `analysis` artifacts from beat agents (for cross-beat context)

**Writes**: `budget-comparison`, `policy-diff`, `trend-note` artifacts to the analysis layer

**Tools**: Same as beat agents — artifact store read, lineage traversal, research search trigger, LLM

**Memory**: **Persistent** — same rationale as beat agents. Research agents maintain:
- Multi-year budget baselines across departments and jurisdictions
- Policy change history (ordinance amendments, regulatory updates)
- Cross-jurisdictional comparisons (is Springfield's budget trend anomalous vs. similar cities?)

**Topology**: Organized by domain rather than beat. Examples:
- `budget-tracker` — monitors budget documents across all departments
- `policy-monitor` — tracks ordinances, regulations, and policy changes
- `public-records` — tracks FOIA filings, court records, regulatory actions

**Key behaviors**:
- Similar to beat agents but oriented toward documents rather than events
- May read beat agent output to add fiscal/policy context to event-driven stories
- The distinction from beat agents is somewhat fuzzy and may collapse in practice — the difference is primarily organizational (one is event-driven, one is document-driven)

## Connection Agent

**Role**: Find non-obvious connections across beats and geographies; surface patterns that no single beat agent would see.

**Reads**: `analysis` artifacts across all beats and geos (async — does not block pipeline); `connections` artifacts (its own prior output)

**Writes**: `connection-memo`, `cross-beat-flag` artifacts

**Tools**:
- Artifact store read with semantic search (primary mechanism — finds thematic similarity across beats)
- Structured query (entity-based: same person or organization appearing across multiple beats)
- Assignment creation (can autonomously trigger research or monitor assignments)
- LLM for cross-domain synthesis

**Memory**: **Persistent** — maintains a working model of ongoing cross-beat threads:
- Known entity appearances across beats (a developer appearing in housing, budget, and city council agendas)
- Developing cross-beat narratives (budget cuts + school closures + council votes as a connected sequence)
- Already-surfaced connections (to avoid re-flagging the same pattern)

**Topology**: Singleton (or small pool). Runs asynchronously — not on the critical path from processing to editorial. Wakes up on a schedule or when a threshold of new analysis artifacts accumulates.

**Key behaviors**:
- Async by design — editor agents do not wait for the connection agent
- Connection artifacts enrich editor agents' input but are not required to produce a draft
- Can autonomously commission research or monitor assignments when a pattern warrants investigation (subject to depth cap and human gate rules from ASSIGNMENTS.md)
- Produces connection memos that are human-readable explanations of the pattern, not just flags

## Editor Agents

**Role**: Produce story drafts from flagged analysis artifacts; iterate based on human feedback.

**Reads**: `analysis` artifacts (beat briefs, flags, trend notes); `connections` artifacts (for cross-beat context); `raw` artifacts (for direct sourcing and quotes)

**Writes**: `story-draft` artifacts

**Tools**:
- Artifact store read (all layers)
- Lineage traversal (to find source quotes and supporting documents)
- LLM for narrative writing
- Style guide / publication standards reference

**Memory**: Light ephemeral per-draft context — knows the editorial feedback it received and what revision it is producing. Does not need persistent memory across stories.

**Topology**: Pool of workers. Triggered when an editor-ready flag accumulates sufficient material, or when a story assignment completes dispatch.

**Key behaviors**:
- Drafts are grounded — every claim should be traceable to a source artifact via lineage
- Receives human editorial feedback via the "send back" flow and produces revised drafts
- A story assignment may direct an editor agent to draft a specific story rather than waiting for organic flag accumulation
- Does not decide what is newsworthy — that judgment belongs to beat agents (flags) and human editorial
- Should surface confidence indicators alongside the draft: which claims have strong sourcing, which have thin coverage

## Assignment Orchestration

Assignments are not a separate agent — they are a coordination protocol executed by the assignment system. But they interact with agents in specific ways worth capturing here:

**What triggers it**: Human editorial, connection agent, beat agent, research agent, or schedule

**What it does**:
1. Queries artifact store (structured + semantic) for existing coverage
2. Runs gap analysis against expected artifact chain
3. Dispatches to agents (beat agents for analysis, research search for missing docs, connection agent for cross-beat coverage)
4. Tracks artifact completion against assignment scope
5. Routes results back to the triggering entity

**Agent interactions**:
- Sends context (assignment ID, scope, query params) to whichever agents it dispatches to
- Agents tag their output with the assignment ID — this is how the collect step works
- Beat and research agents can spawn sub-assignments; connection agent can create top-level assignments

The assignment system acts as a lightweight orchestrator that coordinates agents without owning any analytical capability itself.

## Capability Matrix

| Agent | Persistent memory | Triggers others | Can spawn assignments | Runs on schedule | Runs on demand |
|---|---|---|---|---|---|
| Source examination | — | — | — | — | Yes (developer-triggered; writes spider file) |
| Ingestion worker (Scrapy) | — | — | — | Yes (cron via `spiders due`) | Yes (`spiders run`) |
| Discovery search | Light | Examination | — | Yes | Yes |
| Research search | — | — | — | — | Yes |
| Processing | — | — | — | Reactive | Via assignment |
| Beat agents | **Yes** | Research search | Sub-assignments | Reactive | Via assignment |
| Research agents | **Yes** | Research search | Sub-assignments | Reactive | Via assignment |
| Connection agent | **Yes** | Assignment system | Top-level | Async/periodic | — |
| Editor agents | Ephemeral | — | — | Reactive | Via assignment |

**Reactive** = wakes up when new relevant artifacts appear in the store, not on a fixed schedule.

## Decision log

Record significant design changes here. Keep the doc body current; use this log to explain why it changed.

| Date | Change | Rationale |
|------|--------|-----------|
| 2026-03-18 | Split "Ingestion agents" into Source Examination Agent + Ingestion Worker | Understanding a source and fetching from it are distinct concerns with different triggers and success conditions |
| 2026-03-19 | Examination agent writes Scrapy spider code, not a playbook | Code is deterministic and reviewable; a playbook would require a DSL interpreter or LLM reasoning at every crawl. Spider files are committed to the repo and run without LLM involvement |
| 2026-03-19 | Ingestion worker replaced by deterministic Scrapy spiders | Eliminates LLM cost and non-determinism at crawl time. Playbook-guided agentic fetching was complex to debug and test; a committed spider file is explicit, versionable, and cheap to run |
| 2026-03-19 | Examination agent configured inline, not via `agent_configs` | Examination is a developer-only tool run manually; DB-managed config adds no value for this usage pattern |
| 2026-03-20 | Processing: acquisition + `document-text`; processors only on `raw`+`active` | Matches two-phase raw model; STT and PDF text run only after bytes exist |
